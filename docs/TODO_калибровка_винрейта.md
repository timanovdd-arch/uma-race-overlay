## Фаза 1 — Диагностика (понять, откуда 65/10)

- [ ] **1.1. Лог входного списка скиллов.** В `hooks.rs` (2.1) залогировать массив скиллов
      (id + уровень), который уходит в JSON. Цель — убедиться, что **уники вообще доезжают**
      до сима. Уники в данных игры иногда лежат отдельно от основного массива скиллов —
      если читается только основной, ульты на входе нет.
      → Конкретно: найти в логе id уники Seiun Sky и id уники Fine Motion.

- [ ] **1.2. Per-skill диагностический лог в `sim.rs`.** Для каждого скилла каждой лошади
      за N прогонов вывести:
      - распарсилось условие **честно** или упало в гейт `UNSUPPORTED_COND_P = 0.4` (7.4);
      - **частоту срабатывания** (% прогонов, где скилл сработал);
      - **средний момент** срабатывания (`distance_rate`);
      - **средний вклад** в target speed / accel.
      Это сразу покажет «нечестно стабильные» и «нечестно проваленные» скиллы.

- [ ] **1.3. Прогнать 1.1–1.2 на конкретном составе Fine vs Seiun** и сохранить дамп
      для сравнения до/после правок.

- [ ] **1.4. Режим абляции.** Добавить флаг «выключить скиллы у лошади X», перепрогнать,
      сравнить винрейт.
      - Если без скиллов Fine падает с 65 → ~30 — проблема в **модели скиллов** (фаза 2).
      - Если остаётся ~60 — проблема в **статах/стилях/позиционке** (фаза 3).

## Фаза 2 — Уники и accel-скиллы (главные подозреваемые)

- [ ] **2.1. Уники НЕ проходят Wit-гейт активации.** ⚠️ Ключевая правка.
      Сейчас в 7.3 каждый скилл проходит бросок активации от ума (`1 − 90/Wiz`).
      **Для уник этого броска быть не должно** — ульта срабатывает сразу по выполнению
      условия, всегда. Добавить флаг `is_unique` (из master.mdb) и пропускать Wit-гейт
      для таких скиллов. Иначе ульта Seiun не срабатывает в части прогонов, хотя в игре
      она стабильна.

- [ ] **2.2. Проверить парсинг условия уники Seiun.** Её ульта = accel на входе в спурт
      на первом месте (условия типа `is_lastspurt` / `phase>=3` / `order`). Эти куски в 7.3
      помечены как честно вычисляемые. Убедиться, что **ни один** кусок условия не падает
      в гейт 0.4 (7.4) — иначе самый ценный скилл лошади срабатывает <50% времени.

- [ ] **2.3. То же для уники Fine Motion** — обратная проверка. Если её ульта/скиллы
      проходят через гейт 0.4 там, где в реальности условие жёсткое (и редко выполняется),
      Fine получает завышенный вклад. Перекос может быть с обеих сторон сразу.

- [ ] **2.4. Тайминг accel-эффекта (6.1, 7.3).** Accel-бонус полезен только когда лошадь
      реально разгоняется (current speed < target). Проверить:
      - accel +0.4 у Seiun применяется **в момент входа в спурт** (target прыгает 20→25),
        а не размазывается по случайному окну, где она уже на target speed (тогда буст = 0);
      - для accel-скиллов с `*_random`-условием не выбирается точка, где ускорение
        бесполезно (лошадь на крейсере).

- [ ] **2.5. Длительность эффекта (7.6).** Помечено «не уверен». Для мгновенного accel-рывка
      на спурте длительность сильно решает. Сверить привязку `длительность × дистанция/1000`
      с поведением в игре хотя бы на уровне «бафф ещё активен в нужный момент или уже снят».

## Фаза 3 — Смещение по стилям бега (если фаза 1.4 указала сюда)

- [ ] **3.1. Зафиксировать перекос nige vs sashi/senko (раздел 9).** Отсутствие позиционки
      смещено по стилям, а не симметрично:
      - nige (Seiun) теряет невидимые плюсы лидера (никто не блокирует, Securing the Lead,
        бонус одинокого лидера);
      - sashi/senko (Fine) никогда не получают блокировку — главный реальный риск закрывающих.
      Это, вероятно, занижает Seiun и завышает Fine. Решить: моделировать грубо или
      ввести поправочный коэффициент по стилю как временную меру.

- [ ] **3.2. Свериться, что дистанция трассы определена верно (2.3).** Сейчас восстанавливается
      из maxHP / 2000 м по умолчанию. Неверная дистанция ломает аптитуды, фазы и точку спурта
      сразу у обеих. По возможности достать точную метку из `_simData`.
