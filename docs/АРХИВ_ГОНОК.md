# Архив гонок — продолжение (хэндофф для нового диалога)

Точка входа по фиче «запись всех гонок на диск для офлайн-реверса». Связано с
винрейт-моделью (`docs/WINRATE_MODEL.md`) и блок-анализом
(`docs/БЛОК-АНАЛИЗ_ПРОДОЛЖЕНИЕ.md`).

## Зачем
Чтобы НЕ калибровать винрейт вслепую, копим РЕАЛЬНЫЕ гонки целиком (покадрово) и
реверсим: где скилл дал рывок, где обгон, как HP/блок повлияли, почему победил тот, кто
победил. По этим данным правим модель (`sim.rs`), а не по догадкам.

## Что сделано (плагин, в `main`-дереве, СОБРАНО+ЗАДЕПЛОЕНО 2026-06-17, НЕ закоммичено)
На финише гонки (one-shot `STATS_DONE`, там же где блок-анализ) плагин пишет полный
архив. Файлы:
- `uma-race-overlay/src/archive.rs` (новый) — `write_race(&RaceState, reader)`: meta +
  лошади + таймлайн → JSON, файл.
- `uma-race-overlay/src/frames.rs` — `collect_timeline(reader) -> Option<Timeline>`
  (ещё один проход по кадрам `_simData`, как `compute`): времена кадров + по лошади
  столбцы `d/v/hp/lane/block/temp`. Индекс лошади = индекс кадрового массива (= gate-1).
- `hooks.rs` — вызов `crate::archive::write_race(&race, reader)` в финиш-блоке.
- `writer.rs::json_escape` сделан `pub(crate)`; `lib.rs` += `mod archive`.

## Куда пишет / формат
`%LOCALAPPDATA%\uma_race_overlay_races\race_<ts>_<room|bots>_<instance>.json`
```
{ ts, course_id, track, distance, ground, race_type, instance,
  is_room_match, distinct_trainers,
  frame_t:[...],                       // времена кадров (общие)
  horses:[ { gate,name,trainer,is_user,style,stats[5],apt[5],
             finish_order,finish_time,finish_diff_time,motiv,pop,skills[[id,lvl]],
             d:[...],v:[...],hp:[...],lane:[...],block:[...],temp:[...] } ... ] }
```
- `block` = `BlockFrontHorseIndex` (0xFF свободна), `temp` = TemptationMode (закидывание).
- f32 пишутся `{:.2}`, NaN/inf → `null` (столбцы выровнены по длине с `frame_t`).
- В лог пишется `archive: wrote … KB room/bots`.

## Разделение room match / боты (ПОДТВЕРЖДЕНО живыми гонками 2026-06-17)
ДВА согласованных сигнала (оба в архиве):
- **`race_type`: 8 = room match (PvP), 9 = боты/практика/сюжет.**
- **`distinct_trainers`** (число РАЗНЫХ непустых тренеров): room ≥2 (у NPC тренер пустой).
- Текущий разделитель в коде = `distinct_trainers >= 2`. Если попадётся room с одним
  тренером (вряд ли) — переключить/дополнить на `race_type==8`.
- В имени файла тег `room`/`bots`; instance у room свой (видели 800012 vs ботские 511901)
  → файлы не конфликтуют (плюс таймстамп).

## Подтверждённые цифры (2026-06-17)
- **Боты/практика: 18 лошадей, ~80-90 КБ/гонка.** Room match: **9 лошадей (3 тренера×3),
  ~45 КБ/гонка.** Кадров ~120-140, интервал ~0.76-1.07 c.
- **~100 гонок ≈ ~8-9 МБ.** Диск не проблема.
- Ужать при желании: выкинуть `lane`/`block`/`temp` или хранить только `d/v/hp` (≈ вдвое).

## Как реверсить (для нового чата)
Парсить JSON (питоном/тестом): рывок `v` = сработал скилл (сверить тайминг с distance и
геометрией трассы из course_data.json), смена порядка по `d` между лошадьми = обгон,
просадка `hp` = расход стамины, `block!=0xFF` = денай обгона. Сопоставлять с `skills`
лошади и условиями из master.mdb → понимать, ЧТО решило гонку. Особое внимание: accel
на входе в спурт (где по трассе) — см. инсайты в винрейт-доке.
