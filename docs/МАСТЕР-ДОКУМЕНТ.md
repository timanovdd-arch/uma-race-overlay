# Uma Race Overlay — МАСТЕР-ДОКУМЕНТ

> Один документ, чтобы понять и продолжить проект с нуля (в т.ч. в новом чате).
> Читать сверху вниз. Технические детали кода — в исходниках (`source/`),
> глубокая теория винрейта и план калибровки — в `docs/`.

---

## 0. TL;DR

Оверлей для **Umamusume Pretty Derby** (Steam) поверх мода **Hachimi**.
В PvP-гонке показывает поверх игры таблицу: HP (стамина), скорость, ускорение
каждой лошади, и **%win rate** — экспериментальный прогноз шанса победы через
Monte Carlo симуляцию. Делалось для контента на YouTube (автор — SupperMommy).

Два процесса:
- **плагин** (dll в игре) — только читает данные игры, пишет JSON;
- **приложение** (отдельный exe) — читает JSON, рисует окно, считает симуляцию.

---

## 1. Как пользоваться (для конечного юзера)

Нужно: Steam-версия Umamusume + установленный мод **Hachimi** (или **Hachimi Edge** —
тоже работает, см. §10).

Установка (один раз):
1. `uma_race_overlay.dll` → в папку игры (рядом с UmamusumePrettyDerby.exe).
2. В `hachimi/config.json` добавить `"load_libraries": ["uma_race_overlay.dll"],`
   (или заменить конфиг готовым `config.json` из release).
3. `uma_race_overlay_app.exe` — куда угодно, запускать оттуда.

Каждый раз: запустить игру → запустить exe (при старте донат-окно, Continue) →
зайти в гонку, таблица появится сама.

Хоткеи: **F8** скрыть/показать · **F9** ускорение · **F6** соперники ·
**F10** %win rate (по умолчанию ВЫКЛ — функция в разработке) · **F7** режим мыши.
Окно таблицы перетаскивается мышью (за заголовок) на любой монитор, позиция
запоминается. По пустому месту клики проходят в игру (passthrough).

Свои лошади подсвечены голубым (по имени тренера — подхватывается после первой
гонки против NPC; либо вручную: F6 → клик по своему нику в строке «me:»).

## 2. Архитектура и ПОЧЕМУ так

**Критичный урок:** любой рендер ВНУТРИ процесса игры (hudhook DX11 / egui в
потоке dll) КРАШИТ игру — конфликт за `IDXGISwapChain::Present` с GUI Hachimi.
Поэтому графика и тяжёлые вычисления вынесены в отдельный процесс. Связь —
через JSON-файл `%TEMP%\uma_race_overlay_state.json` (плагин пишет ~20 раз/с,
app читает ~20 раз/с).

```
Игра (+Hachimi)                         Отдельный процесс
┌─────────────────────────┐            ┌──────────────────────────┐
│ uma_race_overlay.dll     │  JSON в    │ uma_race_overlay_app.exe  │
│  - хуки il2cpp (minhook) │ ─%TEMP%──▶ │  - читает JSON            │
│  - читает HorseData      │            │  - egui прозрачное окно   │
│  - пишет state.json      │            │  - Monte Carlo винрейт    │
└─────────────────────────┘            │  - читает master.mdb игры │
                                        └──────────────────────────┘
```

## 3. Сборка и запуск (для разработки)

Требуется Rust (stable, MSVC toolchain) + (для чистой пересборки glfw) CMake.

```
cargo build --release --manifest-path source/uma-race-overlay/Cargo.toml
cargo build --release --manifest-path source/uma-race-overlay-app/Cargo.toml
cargo test  --release --manifest-path source/uma-race-overlay-app/Cargo.toml
```

Перед пересборкой exe убить процесс:
`Get-Process uma_race_overlay_app | Stop-Process -Force`
(иначе cargo не перезапишет файл — os error 5).

Первая сборка glfw тянет CMake/vcvars64; если cargo ругается на glfw —
поставить CMake и VS Build Tools. После первой сборки зависимости закэшированы.

Иконка exe встраивается через `build.rs` (winresource + icon.ico). QR-картинка
донат-окна встроена в exe через `include_bytes!` (donat.png в крейте app).

## 4. Карта файлов

### Плагин `source/uma-race-overlay/src/`
- `lib.rs` — DllMain, запуск фонового потока инициализации.
- `il2cpp.rs` — биндинги il2cpp (всё по именам через GetProcAddress по
  GameAssembly.dll). Хелперы: find_class/method/field, read_string, read_field,
  array_length(+0x18)/array_get_ref(+0x20), class_of, dump_class_methods/fields.
- `hooks.rs` — хуки + ЧТЕНИЕ ВСЕХ ДАННЫХ из HorseData (см. §5). Хуки:
  `Gallop.HorseRaceInfoReplay..ctor(HorseData, RaceSimulateReader)` (identity +
  статы/скиллы лошади, ключ HashMap = указатель this) и
  `HorseRaceInfoReplay.get_RunMotionSpeed()` (покадрово HP/скорость/дистанция).
  Новая гонка = пауза >10с между пачками ctor → clear лошадей.
- `state.rs` — HorseState (модель лошади).
- `writer.rs` — сериализация снапшота в JSON (атомарно: temp+rename).
- `logger.rs` — лог в `%LOCALAPPDATA%\uma_race_overlay.log`.

### Приложение `source/uma-race-overlay-app/src/`
- `main.rs` — весь UI (egui), чтение JSON, донат-окно, дисклеймер win rate,
  иконка окна, запуск симуляции в фоне (`apply_sim`), эвристика-фолбэк
  (`compute_win_chances`), фильтр свои/чужие, перетаскивание окна.
- `gamedata.rs` — чтение `master.mdb` (rusqlite): эффекты/условия скиллов,
  коэффициенты, трассы. Парсер условий-DSL (`parse_cond` → CondTree).
- `sim.rs` — **Monte Carlo симулятор гонки** (ядро винрейта, см. §7).

## 5. Данные

### Из игры (плагин, HorseData) — всё ПОДТВЕРЖДЕНО живой игрой
Геттеры HorseData (имена точные):
- Статы: `get_RawSpeed/RawStamina/RawPow/RawGuts/RawWiz` (с зелёными скиллами!
  get_Speed и т.п. НЕ существует; есть get_Base* — это база без зелёных).
- `get_RunningStyle` (1 nige лидер, 2 senko, 3 sashi, 4 oikomi замыкающий).
- Аптитуды дистанций `get_ProperDistance{Short,Mile,Middle,Long}`, стиля
  `get_ProperRunningStyle{Nige,Senko,Sashi,Oikomi}`, поверхности
  `get_ProperGround{Turf,Dirt}` (1=G..8=S).
- `get_ActiveProperGroundType`/`get_ActiveProperDistance` — аптитуд для
  ФАКТИЧЕСКОЙ поверхности/дистанции этой гонки (игра сама даёт нужный).
- `get_Motivation` (1..5, 3=норма), `get_Popularity` (0..8, меньше=фаворит).
- `get_SkillDataArray` — массив скиллов. Элемент SkillData = ТОЛЬКО 2 поля:
  `skill_id` (+0x10), `level` (+0x14). Эффектов в объекте НЕТ → берём из mdb.
- HP в конструкторе ещё НЕ посчитан (GetHp/GetMaxHp = -1) → HP заполняется в
  кадровом хуке. До старта точная дистанция трассы НЕ известна (см. §8).

JSON-поля на лошадь: gate, name, trainer, hp, max_hp, speed, accel,
max_spurt_accel, distance, spurt, finished, order, style, stats[5], apt[5],
ground[2], adist, aground, gtype, motiv, pop, skills[[id,level]...].

### master.mdb игры (приложение читает при старте)
Путь: `%USERPROFILE%\AppData\LocalLow\Cygames\umamusume\master\master.mdb`
(SQLite, ~13МБ, всегда в синхроне с версией — ничего не хардкодим).
Таблицы:
- `skill_data` (593): эффекты+условия скиллов (см. §7).
- `skill_level_value`: множитель значения эффекта по уровню.
- `race_motivation_rate`: 0.96/0.98/1.0/1.02/1.04 (=±2%/уровень мотивации).
- `race_proper_distance_rate` (скорость, сила по аптитуду дистанции),
  `race_proper_runningstyle_rate` (ум по аптитуду стиля),
  `race_proper_ground_rate` (по аптитуду поверхности).
- `race_course_set`: id→(дистанция, ground 1турф/2грунт). Не привязан к
  текущей гонке (нет course_id из игры).

## 6. Идея винрейта

Это **ПРОГНОЗ**, а не подсмотренный результат. Игра внутри уже считает гонку и
знает победителя (`FinishOrderRawScore`) — это спойлер, НЕ используем (по
требованию автора). Вместо этого САМИ симулируем гонку ~500 раз с разными
случайными бросками (старт, разброс ума, закидывание, срабатывание скиллов) →
доля первых мест = винрейт. Считается в фоне один раз на состав (кэш по hash
состава), не мешая игре.

## 7. Симулятор (`sim.rs`) — полная логика

Шаг времени DT = 1/15 c. Базовая скорость трассы `base = 20 - (D-2000)/1000`.

### 7.1 Подготовка статов (make_runner)
Порядок: `стат - штраф_трассы`, `×мотивация`, `срез >1200 вдвое` (soft_cap:
1500→1350). Ум ещё ×коэф аптитуда стиля. Штрафы трассы (если будет известно
состояние): heavy −50 скорости; turf не-firm −50 силы; dirt не-good −100/good−50;
soft/heavy +6% расхода HP. (Сейчас состояние = firm по умолчанию, см. §8.)
`maxHP = D + 0.8·коэф_стиля·стамина` (коэф: nige .95, senko .89, sashi 1.0,
oikomi .995 — senko хуже всех конвертит, разница ~11%, как в правилах).

### 7.2 Фазы (как в правилах игры)
0 старт (0–1/6), 1 середина (1/6–4/6), 2 終盤 (4/6–5/6), 3 last spurt (5/6–конец).
Last spurt как МЕХАНИКА может начаться раньше (в фазе 2) если хватает HP.

### 7.3 Target speed (к чему стремится скорость каждый кадр)
- Фазы 0–1: `base × коэф_стиля_фазы` (Speed-стат и аптитуд дистанции тут НЕ
  влияют — так в правилах). +закидывание ×1.6 в «своей» секции.
- Фаза 2 / спурт: `base×коэф + вклад_Speed + вклад_Guts` (+ в спурте ×1.05
  и доп. слагаемые). вклад_Speed = √(500·spd)·аптДист·0.002; вклад_Guts =
  (450·guts)^0.597·0.0001.
- + разброс ума по 24 секциям + скилл-бонусы скорости.
- **Позиционные поправки (3.1):** одинокий лидер-nige (order==1, phase≥1) ×1.012
  (Securing the Lead); «упёршийся» закрывающий в спурте (order>3) ×0.93
  (блокировка — из правил «especially fatal for backline»).

### 7.4 Ускорение и HP
accel = 0.0006·√(500·pow)·коэф_стиля·коэф_поверхности·коэф_дистанции
+ скилл-accel; если v<0.85·base → +24 стартовый рывок. Тормоз −0.8..−1.2 по фазе.
Расход HP = 20·(v−base+12)²/144; в спурте ×(1+200/√(600·guts)); закидывание ×1.6;
soft/heavy грунт ×1.06. HP=0 → ползёт на минималке.
plan_spurt: в фазе 2 линейно считает, на сколько метров спурта хватит HP →
спурт начинается раньше/позже (как в правилах «не хватает HP → урезается/сдвиг»).

### 7.5 Скиллы — главное
Из mdb: до 2 вариантов, в каждом до 3 эффектов + условие. Значение =
значение_БД × множитель_по_уровню. Типы эффектов: **27 current speed**,
**31 acceleration**, **9 recovery** (% от maxHP), 21 target speed, 8/13 пассивы
старта/закидывания, **1–5 зелёные — ПРОПУСКАЕМ** (уже в Raw-статах), дебаффы
(target_type≠1 — на всех соперников). Прочие (обзор/дорожки) не моделируем.
- **Wit-гейт активации:** каждый скилл раз за забег проходит бросок с шансом от
  ума (`1−90/wiz`). **ИСКЛЮЧЕНИЕ: СВОЙ уник** персонажа (固有, skill_category==5 И
  id<200000, т.е. 1xxxxx) — БЕЗ гейта, всегда по условию. **Наследуемые уники
  (継承固有, id 9xxxxx) — гейт ЕСТЬ** (они слабее, проходят как обычные). См.
  `passes_activation_gate` + `is_unique` в gamedata.
- **Условия-DSL** (`a&b@c` = (a И b) ИЛИ c): честно вычисляются phase, order,
  order_rate, distance_rate, remain_distance, hp_per, running_style,
  distance_type, ground_type, is_lastspurt, is_finalcorner, corner, accumulatetime,
  post_number, popularity. `*_random` → случайная точка в окне дистанции.
  Углы СИНТЕТИЧЕСКИЕ (геометрии трасс нет). Непросчитываемые позиционные условия
  (blocked_*, bashin_diff_*, is_overtake, change_order_*) → вероятностный гейт
  `UNSUPPORTED_COND_P=0.4`.
- accel-скилл полезен только когда v<target (на входе в спурт target прыгает
  20→25) — в модели accel складывается в accel только в фазе разгона. ✔

### 7.6 Честность RNG
Каждой лошади — СВОЙ независимый RNG (сид из общего потока). Раньше число
скиллов одной лошади сдвигало броски другой → A/B был кривой. Исправлено.

## 8. Что НЕ домоделировано / заглушки (влияет на точность)

- **Точная дистанция трассы**: до старта берётся дедукцией из maxHP или 2000м по
  умолчанию. Точную метку надо достать из `_simData` внутри RaceSimulateReader
  (дамп структуры уже есть в логе: поля _simData(+0x10) и методы GetDistance,
  GetLastSpurtStartDistance, GetLastFrameDistance — отсюда тянуть).
- **Состояние грунта (firm/good/soft/heavy) и погода**: сейчас firm/ясно по
  умолчанию. Они per-race, лежат в `_simData` (в master.mdb их нет). Штрафы под
  них УЖЕ заложены в sim (RaceParams.condition) — нужно лишь прокинуть значение.
- **Пороги статов трассы** (бонус скорости за высокий стат): не учтены (нужен
  course_id; race_course_set_status даёт какой стат-порог, но не значение).
- **Позиционные механики** (грубо/опущены, требуют симуляции дорожек):
  Position Keep (6 режимов), блокировка соперником, Spot Struggle, Dueling,
  Repositioning, Securing the Lead, Stamina Contest, Power Conservation, уклоны.

## 10. Дистрибуция

`release/` — то, что раздаётся (= UmaRaceOverlay.zip):
`uma_race_overlay_app.exe`, `uma_race_overlay.dll`, `config.json` (готовый
Hachimi-конфиг с load_libraries + auto_full_screen=false), `README.txt` (англ.,

нужны; можно переопределить, положив donat.png / donate_link.txt рядом с exe).

зависит от внутренностей Hachimi (нужно лишь чтобы его загрузили в процесс).
Для Edge лучше НЕ заменять их config.json нашим (схема может отличаться) —
дописать строку load_libraries в их конфиг. Конфликта по рендеру нет (мы в
отдельном процессе).

## 11. Грабли (чтобы не наступить снова)

- il2cpp трогать только ПОСЛЕ загрузки `cri_ware_unity.dll` (иначе краш). Сборку
  получать `il2cpp_domain_assembly_open(domain,"umamusume.dll")`, НЕ перебором.
  Адреса — через GetProcAddress по GameAssembly.dll.
- Рендер/нагрузка ВНУТРИ процесса игры = краш (см. §2).
- PowerShell 5.1 читает `.ps1` как ANSI → не клади кириллицу в скрипты.
- serde_json (Hachimi) НЕ принимает BOM → конфиги писать UTF-8 без BOM.
- Запуск игры только `steam://rungameid/3224770` (прямой exe сегфолтит — DRM).
- GLFW ставит окну свою дефолтную иконку → иконку панели задач ставим в рантайме
  (set_window_icon через set_icon_from_pixels), не только в exe-ресурсе.
- egui::Window с `.anchor(...)` НЕ перетаскивается → использовать `.default_pos`.
- Тестировать app без игры: писать фейковый state.json (ts=epoch_ms свежий,
  running:true). НЕЛЬЗЯ когда юзер в игре/за компом (дерётся с живым плагином).
- Лог плагина: `%LOCALAPPDATA%\uma_race_overlay.log`, успех = `race hooks installed`.
