# Фидбэк (баг/идея) + защита — хэндофф для нового диалога

Точка входа для отдельного диалога по системе обратной связи оверлея и её защите.
Общая картина проекта — `МАСТЕР-ДОКУМЕНТ.md`. Блок-анализ — `БЛОК-АНАЛИЗ_ПРОДОЛЖЕНИЕ.md`
(не путать, это разные темы).

## Что это и зачем
Кнопка **«✉ Feedback»** в оверлее шлёт разработчику баг или идею (опц. с приложенной
гонкой — для крауд-сбора блок-кейсов). Чтобы был настоящий rate-limit/чёрный список и
чтобы НА РАЗРАБОТЧИКА НЕЛЬЗЯ БЫЛО ВЫЙТИ, отправка идёт НЕ напрямую в Telegram, а через
релей: **app → Cloudflare Worker → Telegram**. Токен бота и chat_id живут ТОЛЬКО в Worker.

## Архитектура (поток)
1. App собирает поля (анонимный `install_id`, kind=bug/idea, text, contact, areas,
   toggles, app_version, os, опц. файл гонки) и шлёт multipart-POST на URL Worker
   (заголовок `X-App-Key`). HTTP делает встроенный `curl.exe` (без TLS-зависимостей).
2. Worker: проверка app-key → защита (см. ниже) → пересборка текста → пересылка в
   Telegram (`sendMessage` + `sendDocument` для гонки). Отвечает JSON+HTTP-кодом.
3. App мапит код: 200→Sent, 429→RateLimited(мин), 403→Blocked, иначе→Failed.

## Защита (всё в `feedback-relay/worker.js`, server-side)
- **Rate-limit до 25 сообщений / сутки** (UTC-день) на `install_id` (KV `rl:<id>:<YYYY-MM-DD>`
  счётчик, TTL 26ч; `RL_MAX_PER_DAY`=25). Превышение → 429 (retry_after = до полуночи UTC).
- **Авто-бан**: `VIO_THRESHOLD`=5 превышений → бан `install_id` на `BAN_TTL_S`=7 дней (`bl:`).
- **Чёрный список** `install_id` и хэша IP (`bl:`/`blip:`) → 403, сообщение игнорируется.
- **Вторичный IP-лимит** `IP_MAX_PER_WINDOW`=30/час (`ipc:`) — анти-ротация install_id.
- Все пороги — КОНСТАНТЫ вверху `worker.js`. Поменял → передеплой воркера (см. ниже).
- Производительность: Cloudflare KV, ключи с TTL; тянет тысячи (до 25 запросов/сутки на юзера).
  KV eventually-consistent (под бурстом одного юзера пара лишних может проскочить — для
  анти-спама ок; строгая атомарность → Durable Objects, отдельная доработка).

## Приватность (требование «данных нет / не выйти на меня»)
- Токена бота и chat_id в КЛИЕНТЕ НЕТ → через exe на разработчика не выйти.
- `install_id` — случайные 128 бит (hex32), БЕЗ PII; имени/почты не собираем.
- IP НЕ хранится сырым — только солёный SHA-256-хэш (соль `IP_SALT`) и с TTL; в Telegram
  IP не пересылается. Все ключи кроме бана протухают по TTL.
- Полный `install_id` пишется в сообщение (строка `ID: …`) — нужен для ручного бана.

## Карта файлов
### Релей `feedback-relay/`
- `worker.js` — Worker (ES-module `export default {fetch}`). Защита + пересылка. Формат
  сообщения собирает `composeMessage` ЗДЕСЬ (меняется без пересборки app).
- `wrangler.toml` — для CLI-деплоя (необязателен: деплоили через дашборд). KV id — плейсхолдер.
- `README.md` — инструкция деплоя (дашборд И wrangler), ротация токена, ручной бан.
- `secrets.local.txt` — **GITIGNORED**: CHAT_ID/APP_KEY/IP_SALT (+ напоминание про BOT_TOKEN).
### Клиент `uma-race-overlay-app/`
- `src/feedback.rs` — `RelayConfig{url,app_key}`, `Report{...}`, `SendStatus{Idle,Sending,Sent,
  RateLimited(min),Blocked,Failed}`, `spawn_send` (фоновый curl multipart, `--form-string` для
  текста, `-F race=@tmpfile` для гонки, `-w %{http_code}` + тело в темп-файл → `map_response`).
  URL/key читаются из `env!("UMA_RELAY_URL")`/`env!("UMA_RELAY_KEY")` (подставляет build.rs).
- `src/main.rs` — поля App `relay_cfg`, `install_id`, `fb_*`, `fb_status`; `new_install_id()`;
  `Cfg.install_id` (хранится в `%TEMP%\uma_race_overlay_cfg.json`); UI меню Bug/Idea + форма
  (галочки `FB_AREAS`, текст, attach гонки, контакт), статусы (RateLimited→«wait ~N min»,
  Blocked→нейтрально). `fb_send` собирает `Report` и зовёт `feedback::spawn_send`.
- `build.rs` — `embed_relay_config()`: читает gitignored `feedback_creds.txt` (или env
  `UMA_RELAY_URL`/`UMA_RELAY_KEY`) → `cargo:rustc-env` + `rerun-if-changed`. Так секрет НЕ
  попадает в публичный исходник, но локальные сборки подхватывают автоматически.
- `feedback_creds.txt` — **GITIGNORED**: `relay_url=…` + `relay_key=…` (=APP_KEY).

## Как делать частые операции
### Поменять лимиты / логику Worker
1. Правишь `feedback-relay/worker.js` (константы вверху или код).
2. Дашборд → Worker `uma-feedback` → **Edit code** → вставить новый код целиком → **Deploy**.
   (Если Deploy серый: внеси правку пробел+Backspace или Ctrl+S, чтобы редактор увидел изменение.)
3. Проверка: `curl https://uma-feedback.timanov-d-d.workers.dev/` → `{"status":"ok"}`.
### Тест канала curl'ом (PowerShell)
```
$u="https://uma-feedback.timanov-d-d.workers.dev/"; $k="<APP_KEY из secrets.local.txt>"
curl.exe -s -X POST $u -H "X-App-Key: $k" --form-string "install_id=test_xxx" `
  --form-string "kind=idea" --form-string "text=test" --form-string "app_version=0.2.0" `
  --form-string "os=windows" -w "`nHTTP %{http_code}"
```
Ожидание: 200 + сообщение в Telegram. 26-й раз тем же id за сутки → 429.
### Ручной бан / разбан (через дашборд KV или wrangler)
Полный `install_id` виден в сообщении (`ID: …`). Дашборд: Workers KV → `uma-feedback` → KV Pairs →
добавить ключ `bl:<install_id>` = `1`. Или wrangler: `wrangler kv key put bl:<id> 1`.
### Ротация токена (закрыть утёкший в v0.2.0)
Теперь БЕЗОПАСНО в любой момент (новый exe токена не содержит): @BotFather `/revoke` → новый токен
→ обнови секрет `BOT_TOKEN` в воркере. Релей продолжит работать; старые v0.2.0 (прямой Telegram) сломаются.
### Пересобрать клиент (после правок Rust)
```
Get-Process uma_race_overlay_app | Stop-Process -Force   # иначе os error 5
cargo build --release --manifest-path uma-race-overlay-app/Cargo.toml
cargo test  --release --manifest-path uma-race-overlay-app/Cargo.toml
```
URL/key подхватятся из `feedback_creds.txt` автоматически. Деплой exe → `UmaRaceOverlay-RELEASE\`.

## Грабли (подтверждены при настройке 2026-06-15)
- **Дашборд: кнопка Deploy сначала серая** — внеси любую правку (пробел+Backspace) или Ctrl+S.
  Превью-панель справа не нужна (часто ругается «Improperly configured Workers Preview» из-за
  блокировки cookies — игнорировать, проверять реальный URL).
- **KV-binding добавляется в Settings ВОРКЕРА** (Compute → Workers → uma-feedback → Settings →
  Bindings), НЕ в настройках самого KV namespace.
- **Диагностика ошибок POST**: `1101`/500 = нет binding `STORE`; `forward_failed`/502 = неверный/
  отсутствует `BOT_TOKEN`/`CHAT_ID`; `401` = не совпал `APP_KEY`; `429`/`403` = это и есть защита.
- **Клиент: `option_env!` через cargo НЕНАДЁЖЕН** (флаки-перелинк) → секреты вшиваем через build.rs +
  `cargo:rustc-env` + `rerun-if-changed`. **Байт-греп exe на строку врёт** при opt-level=2 —
  проверять рантайм (`configured()`), а не наличие байтов.
- Печатать текст в форме — держа курсор над окном (passthrough иначе уводит клавиши в игру; F7 — форс).
