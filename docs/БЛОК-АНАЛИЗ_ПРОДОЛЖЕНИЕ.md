# Блок-анализ — продолжение разработки (хэндофф для нового диалога)

Этот файл — точка входа для отдельного диалога, посвящённого блок-анализу.
Этот док самодостаточен (реверс/история сведены сюда). Общая картина — в
`МАСТЕР-ДОКУМЕНТ.md`. Винрейт-модель (вкл. калибровку) — в `WINRATE_MODEL.md` (англ.).
Запись реальных гонок на диск (покадрово, для реверса) — `docs/АРХИВ_ГОНОК.md`.

## Цель
Довести «блок-анализ реальной гонки» до точного, которому можно доверять, показателя
влияния блока на исход + красивый визуал для контента. Базовая версия уже работает и
зарелизена (v0.2.0, experimental) для крауд-теста — теперь калибровка и доводка.

## Важные границы
- Метрики блока (mid leg / spurt block) строятся на РЕАЛЬНЫХ данных гонки (frames.rs).
- `sim.rs` (Monte Carlo) — отдельная тема (винрейт-чат). ИСКЛЮЧЕНИЕ: «place w/o block»
  теперь использует симулятор как контрфактуал — можно править трафик-тумблер
  (`block_model`/`simulate_no_block`), но КАЛИБРОВКУ модели вести в винрейт-чате.
- Реверс структур ЗАВЕРШЁН — оффсеты найдены и подтверждены живой игрой (см. ниже).
  Повторно дампить классы НЕ нужно.

## С чего начать новому диалогу
1. Прочитать этот файл (самодостаточен).
2. Запросить у юзера свежие скрины/логи гонок тестеров с блоком (для валидации).
3. Лог плагина: `%LOCALAPPDATA%\uma_race_overlay.log`, строки `frame stats:`.

## Текущее состояние (готово, в `main`, коммит про block analysis)
- Плагин на ФИНИШЕ гонки один раз проходит предрассчитанные кадры из `_simData`
  (`frames.rs::compute`) и считает per-horse: блок (любой фазы), блок ДО спурта,
  kakari, и отдельно **блок в спурте** + оценку потерянного времени/дистанции. JSON.
- App показывает (чекбокс «block analysis») 3 колонки:
  - **«mid leg block»** — суммарный фронт-блок ДО начала спурта (序盤+中盤), Xs;
    `clean` если не было. Контекст «насколько затолкан был заезд», не штраф.
  - **«spurt block»** (Xs ×N) — блок в спурте (реально душит ускорение).
  - **«place w/o block»** — контрфактуальное место `P5→P2` ИЗ СИМУЛЯТОРА (см. правки
    2026-06-17); дельта мест в тултипе.

## Правки 2026-06-15 (сессия калибровки, после релиза v0.2.0)
Все правки собраны и задеплоены (DLL в игру + RELEASE, exe в RELEASE). НЕ закоммичено
в git — следующий чат может закоммитить. Файлы менялись: `frames.rs`, `hooks.rs`,
`state.rs`, `writer.rs` (плагин) и `main.rs` (app).

1. **Новая колонка «mid leg block»** = `pre_spurt_blocked_time`. Суммарный фронт-блок
   ДО начала спурта (序盤+中盤), в секундах; `clean` если не было. Поле во всей цепочке
   `Accum`→`HorseState`→JSON→`Horse`; копится в `frames.rs`, пока `is_blocked && !in_spurt`.
   В UI это ПЕРВАЯ из трёх блок-колонок (block-колонок стало 3). Лог: `preSpurtBlk X.XXs`.

2. **Фикс `max spurt accel`** (видели 150/116 м/с² — это был шум, не ускорение).
   Причина: `accel = Δскорость/Δt`, где `Δt` — гэп между вызовами хука, бывает <1мс →
   деление на почти-ноль → спайк, который «липко» застревал в `max`. Живая колонка
   `accel` его не показывала (он затухал за пару кадров), а `max` — да.
   ВАЖНО (ground truth от юзера): живая `accel` КОРРЕКТНА — ульта Seiun даёт 0.4 (база)/
   0.2 (передаваемая). Значит реальный масштаб ускорения ~0.x, а 20-50 — всегда был шум.
   Решение (НЕ трогая поведение живой колонки): считать ускорение по **фиксированному
   окну ~50мс через якорь** (`accel_anchor_speed`/`accel_anchor_time` в `HorseState`),
   а не по гэпу вызовов. `Δt` всегда осмысленный, частота вызова хука не важна (это
   ломало прошлую попытку с порогом 8мс → колонка показывала «—»). Кламп сырого
   `±15 м/с²` как страховка. Установившееся 0.4 сохраняется; `max spurt accel` стал ~0.4.

3. **Фикс `place w/o block` (бот «P15→P2»)** — реальный баг трактовки поля.
   `RaceSimulateHorseResultData +0x1c FinishDiffTime` — это отрыв до лошади на ОДНО
   МЕСТО впереди (margin), а НЕ до победителя (доказано логом: 15-е место имело
   `diff 0.010`, а 4-е `0.197` < 2-е `0.703` — невозможно для кумулятивного отрыва).
   Контрфактуал в `main.rs` сортировал по `finish_diff_time − spurt_lost_time` → 15-е
   место с margin 0.01 улетало в начало. Исправлено: читаем **абсолютное `FinishTime`
   (`+0x14`)** в `Accum.finish_time`→JSON→`Horse`, и cf-ранжирование + `gap_ahead`
   теперь по `finish_time`. Лог: добавлен `finishT {:.3}`.
   **TODO-валидация (следующий чат):** убедиться по логу, что `finishT` — это абсолютное
   время финиша (~90-110с, растёт от 1-го места к последнему). Если там не секунды —
   подобрать оффсет (`+0x14 FinishTime` vs `+0x18 FinishTimeRaw`).

## Правки 2026-06-17 — «place w/o block» через симулятор (block kill)
После того как винрейт-чат закончил улучшение симулятора (Position Keep = Pace Down,
трафик-модель `BLOCK_DIST`/`pass_prob`/`BLOCK_LEAK`, реальная геометрия трасс), по
решению юзера КОНТРФАКТУАЛ блока переведён на симулятор. **Только app** (`sim.rs`+`main.rs`),
DLL не трогали; собрано, 22 теста зелёные, exe+DLL обновлены в RELEASE.
- `sim.rs`: флаг `block_model: bool` протянут через `run_race`/`accumulate`/
  `simulate_with_workers`; трафик-гейт под ним. Новый `pub fn simulate_no_block(...)` —
  тот же сид (парное A/B), но без блок-модели. `simulate`/`simulate_mode` шлют `true`.
- `main.rs apply_sim`: второй прогон `simulate_no_block`; `block_kill = avg_place(с блоком)
  − avg_place(без)` на лошадь; кэш `sim_slot` теперь хранит `(win, block_kill)`. Фоном,
  2×3000 прогонов (~0.9с/6 ядер), один раз на состав.
- `main.rs gui_run`: `cf_place` теперь = `rank − block_kill`, пересортировка поля (была
  эвристика finish_time). Поле `Horse.block_kill` (serde default = NaN через `nan_f32`).
- UI: колонка **«place w/o block»** = `P{rank}→P{cf}` (red топ-1 / orange выше / green
  ниже-выигрыш / gray без изм.), точная дельта мест в тултипе. Отдельная колонка «block
  kill» НЕ заводилась (схлопнута сюда). Блок-колонок снова 3.
- Граница: блок-чат теперь МОЖЕТ трогать `sim.rs` в части трафик-модели/тумблера для
  этого контрфактуала; калибровка самой модели — по-прежнему в винрейт-чате.

## Оффсеты (подтверждены живой игрой 2026-06-14)
- `RaceSimulateReader` +0x10 `_simData`.
- `RaceSimulateData` +0x18 `_frameDataList`(List), +0x20 `_horseResultDataArray`;
  List backing array `_items` в +0x10.
- `RaceSimulateFrameData` (class): +0x10 `Time`(f32), +0x18 `HorseDataArray`.
- `RaceSimulateHorseFrameData`: +0x10 `Distance` +0x14 `LanePosition` +0x18 `Speed`

## Метрика сейчас
### Спурт-блок (по реальным кадрам) — колонки «mid leg block» / «spurt block»
- `pre_spurt_blocked_time` = суммарный фронт-блок ДО спурта (контекст, не штраф).
- `spurt_blocked_time` + оценка `spurt_lost_*` по «скорости освобождения»: на снятии
  блока опора = `max(скорость входа, скорость сразу после освобождения)`; потеря =
  `∫(опора − v)·dt`. Это РЕАЛЬНЫЕ данные гонки (frames.rs), остаются как есть.

### «place w/o block» — теперь КОНТРФАКТУАЛ ИЗ СИМУЛЯТОРА (2026-06-17)
- Старую эвристику (вернуть спурт-потерю и пересортировать finish_time) **заменили**.
- Теперь: `apply_sim` гоняет реальный состав через винрейт-симулятор ДВАЖДЫ — с блок/
  трафик-моделью и без (`sim::simulate` vs `sim::simulate_no_block`, тот же сид = парное
  A/B). `block_kill[i] = avg_place(с блоком) − avg_place(без)` = на сколько мест трафик
  сдвинул КАЖДУЮ лошадь. В `gui_run`: реальному месту возвращаем этот сдвиг
  (`score = rank − block_kill`), пересортировываем поле → `cf_place`. UI: ячейка
  `P{rank}→P{cf}` (красный=в топ-1, оранжевый=выше, зелёный=ниже/выиграл от зажатых
  соперников, серый=без изменений), точная дельта мест — в тултипе.
- Это решило главную слабость старого контрфактуала: **реинтеракция** (без блока позиции
  реально переигрываются в симуляции, а не просто возврат времени).
- ВАЖНО: это МОДЕЛЬНАЯ оценка (среднее по полю, не привязана к RNG конкретной гонки) и
  её точность упирается в калибровку симулятора (см. винрейт-чат). Отдельной колонки
  «block kill» нет — схлопнута в «place w/o block».

### Остаточные слабые места (кандидаты на доводку)
1. **gate8-эффект** (spurt_lost по реальным кадрам): едет ровно за спиной → дефицит ≈ 0
   → потеря занижена, хотя ускорение задушено. (Касается spurt_lost, не cf_place.)
2. Точность `cf_place` = точность симулятора (пороги трафика/позиционка ещё калибруются).
3. Только ФРОНТ-блок (`BlockFrontHorseIndex`); side-блок отдельного поля в кадре нет.
