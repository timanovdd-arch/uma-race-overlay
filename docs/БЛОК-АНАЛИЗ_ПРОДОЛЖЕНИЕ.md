# Блок-анализ — продолжение разработки (хэндофф для нового диалога)

Этот файл — точка входа для отдельного диалога, посвящённого ТОЛЬКО блок-анализу.
Полный реверс и история — в `РАЗБОР_БЛОКА_И_ПОРТРЕТЫ.md`. Общая картина проекта —
в `МАСТЕР-ДОКУМЕНТ.md`. Теория винрейт-калькулятора — в `ЛОГИКА_ВИНРЕЙТА.md`.

## Цель
Довести «блок-анализ реальной гонки» до точного, которому можно доверять, показателя
влияния блока на исход + красивый визуал для контента. Базовая версия уже работает и
зарелизена (v0.2.0, experimental) для крауд-теста — теперь калибровка и доводка.

## Важные границы
- Это НЕ про калькулятор винрейта (Monte Carlo). Файл `uma-race-overlay-app/src/sim.rs`
  НЕ трогать — отдельная тема. Здесь всё строится на РЕАЛЬНЫХ данных гонки.
- Реверс структур ЗАВЕРШЁН — оффсеты найдены и подтверждены живой игрой (см. ниже).
  Повторно дампить классы НЕ нужно.

## С чего начать новому диалогу
1. Прочитать этот файл + `РАЗБОР_БЛОКА_И_ПОРТРЕТЫ.md`.
2. Запросить у юзера свежие скрины/логи гонок тестеров с блоком (для валидации).
3. Лог плагина: `%LOCALAPPDATA%\uma_race_overlay.log`, строки `frame stats:`.

## Текущее состояние (готово, в `main`, коммит про block analysis)
- Плагин на ФИНИШЕ гонки один раз проходит предрассчитанные кадры из `_simData`
  (`frames.rs::compute`) и считает per-horse: блок (любой фазы), kakari, и отдельно
  **блок в спурте** + оценку потерянного времени/дистанции. Пишет в JSON.
- App показывает (чекбокс «block analysis»): колонку **«spurt block»** (Xs ×N) и
  **«place w/o block»** — контрфактуальное место `P5→P2` (+ `?` если зажат до финиша).

## Технические якоря (файлы)
- `uma-race-overlay/src/frames.rs` — ЯДРО расчёта (один проход по кадрам).
- `uma-race-overlay/src/hooks.rs` — вызов `frames::compute` на финише (все finished,
  one-shot `STATS_DONE`); `RACE_READER` стэшится в ctor, сброс на новой гонке.
- `uma-race-overlay/src/state.rs` + `writer.rs` — поля HorseState + сериализация в JSON.
- `uma-race-overlay/src/il2cpp.rs` — хелперы: `array_element_class`, `is_valuetype`,
  `array_element_size`, `array_elem_base` (struct/ref-массивы).
- `uma-race-overlay-app/src/main.rs` — поля `Horse` (serde), расчёт `gap_ahead`/`cf_place`
  в `gui_run` (по ВСЕМУ составу до фильтрации), рендер в `race_table`.

## Оффсеты (подтверждены живой игрой 2026-06-14)
- `RaceSimulateReader` +0x10 `_simData`.
- `RaceSimulateData` +0x18 `_frameDataList`(List), +0x20 `_horseResultDataArray`;
  List backing array `_items` в +0x10.
- `RaceSimulateFrameData` (class): +0x10 `Time`(f32), +0x18 `HorseDataArray`.
- `RaceSimulateHorseFrameData`: +0x10 `Distance` +0x14 `LanePosition` +0x18 `Speed`
  +0x1c `Hp` +0x20 `TemptationMode`(u8) +0x21 `BlockFrontHorseIndex`(u8; 0xFF=свободна).
  МОЖЕТ быть value-типом → читать с поправкой `adj=0x10` (см. `frames.rs`).
- `RaceSimulateHorseResultData`: +0x10 `FinishOrder`(0-based) +0x14 `FinishTime`
  +0x18 `FinishTimeRaw` +0x1c `FinishDiffTime` +0x28 `LastSpurtStartDistance`
  +0x2c `RunningStyle` +0x30 `Defeat`.
- **Маппинг:** индекс кадрового массива = `gate-1` (подтверждено `simFO == liveFO-1`).

## Метрика сейчас (и её слабые места — главные кандидаты на доводку)
- `spurt_lost_*` оцениваются по «скорости освобождения»: на снятии блока опора =
  `max(скорость входа, скорость сразу после освобождения)`; потеря = `∫(опора − v)·dt`.
- Контрфактуал «place w/o block»: вернуть КАЖДОМУ его спурт-потерю времени и
  пересортировать `finish_diff_time` (`ct = finish_diff_time − spurt_lost_time`).
- Слабые места:
  1. **gate8-эффект:** если лошадь весь блок едет ровно за спиной (не сбрасывает
     скорость), дефицит ≈ 0 → потеря занижена, хотя ускорение реально задушено.
  2. **spurt_unresolved** (зажат до финиша): свободную скорость из одной гонки не узнать
     → занижено, помечаем `?`. Возможный мостик — стат-потенциал (опц., аккуратно).
  3. Контрфактуал не учитывает РЕИНТЕРАКЦИЮ (без блока позиции переиграются), только
     возврат времени. Это допущение — честно подписать/улучшить.
  4. Опорная скорость грубая; кандидат — кап по скорости соперника впереди / по
     target-speed фазы спурта.
  5. Сейчас только ФРОНТ-блок (`BlockFrontHorseIndex`). Side-блок отдельного поля в
     кадре не имеет — вывести из `LanePosition`+соседи, если нужно.
